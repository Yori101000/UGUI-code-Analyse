# 第15章 文本渲染系统

> 本文是对原文的结构化重写。原文 15.1、15.2、15.3 中"动态图集→textureRebuilt→SetAllDirty→Canvas 级联刷新"同一链路重复了三次以上，15.5 排版系统 18 个小节深入字体排印通识但与 UGUI 机制关联较弱。现合并为 6 节，按初学者路径自底向上：字符→图集→动态更新→Text 渲染流程→TMP→优化。对照 UGUI 源码修正了 5 处错误/不准确表述。

---

## 概述

文本渲染和 Image 渲染本质上的不同：

```
Image 渲染：固定 Sprite → 固定 UV → 一个 Quad（4 顶点）→ GPU
Text 渲染：字符串 → 逐个字符查字体 → 动态生成 Quad 网格 → GPU
```

Image 固定 4 个顶点，Text 每多一个字符就多 4 个顶点。100 个字的文本 = 400+ 顶点，这就是为什么大量文本比大量 Image 更消耗 CPU。

---

## 15.1 从字符到像素：文本渲染的完整链路

### 15.1.1 两个基础概念

**字符（Character）≠ 字形（Glyph）**

- **字符**是逻辑层面——Unicode 编码，如 `U+0041` 代表字母 "A"。它只告诉你"这是什么符号"，不包含图形信息。
- **字形（Glyph）**是视觉层面——"这个字符在当前字体下长什么样"。同一个字符 "A" 在宋体、黑体、Arial 里看起来都不一样。

文本渲染系统操作的是 Glyph，不是 Character。它的工作：拿到字符串 → 逐字符查对应 Glyph → 找到该 Glyph 在图集中的 UV → 生成 Quad。

**字体文件（.ttf/.otf）≠ 字形图集（Font Atlas）**

- **字体文件**保存的是矢量曲线（贝塞尔曲线），描述每个字符的轮廓。分辨率无关，放大不会模糊。
- **字形图集**是 GPU 真正使用的纹理。GPU 不能渲染贝塞尔曲线，需要先把矢量轮廓转成像素——这个过程叫**栅格化（Rasterization）**，由操作系统底层的 FreeType 库完成，不是 Unity C# 代码直接做的。

类比：字体文件是"建筑图纸"（矢量，可任意缩放），字形图集是"拍好的照片"（像素，固定分辨率）。渲染时 GPU 用的是照片。

### 15.1.2 字形图集：一张纹理放所有字符

GPU 不希望每渲染一个字符就切换一次纹理（会打断 Batch）。系统把多个字符的栅格化结果打包到同一张纹理——**字形图集（Glyph Atlas）**：

```
┌──────────────────────────┐
│  [A]  [B]  [C]  [D]     │
│  [E]  [F]  [中] [文]    │
│  [你] [好]              │
└──────────────────────────┘
```

每个字符在图集中占据一块矩形区域。渲染 "A" 时，系统用 UV 坐标告诉 GPU："去图集左上角采样那块"。一块图集纹理 = 所有字符共享 = 可以合批。

**Advance（前进距离）**：渲染完一个字符后，光标往右移动多少像素才画下一个字符。比如 "A" 的 Advance=20，"B" 的 Advance=18。文本排版本质上就是不断累加 Advance：`cursorX += glyph.advance`。

### 15.1.3 静态 vs 动态字体图集

| | 静态图集 | 动态图集 |
|------|------|------|
| 何时生成 | 编辑阶段一次性烘焙 | 运行时按需生成 |
| 字符范围 | 预先指定（如 ASCII 全集） | 用到什么生成什么 |
| 运行时性能 | 稳定，无额外开销 | 首次遇到新字符时卡顿 |
| 内存 | 固定、可预估 | 随使用增长 |
| 适合场景 | 英文项目、字符集固定 | 中文项目、聊天系统 |

**中文字符数以万计**——GB2312 6000+ 字，CJK 全集数万字。全部预烘焙到一张图集会超过硬件纹理尺寸限制，所以中文项目必然用动态图集。

**动态图集的致命问题：纹理空间是有限的。** 图集满了（如 1024² 纹理装不下更多字），系统面临选择：**扩容**（512→1024→2048，显存增加）或**重建**（重新排列字符位置 → 所有旧 UV 失效）。无论哪种，纹理内容变了，就会触发级联反应。

## 15.2 Font.textureRebuilt：文本系统最关键的性能链路

一次图集更新，影响所有使用该字体的 Text 组件：

```
字体图集纹理发生变化
  ↓
Font.textureRebuilt 事件触发
  ↓
FontUpdateTracker 找到所有引用该字体的 Text
  ↓
每个 Text.FontTextureChanged()
  ↓
SetAllDirty() → Layout Dirty + Vertex Dirty + Material Dirty
  ↓
下一帧 CanvasUpdateRegistry 统一执行 Rebuild
  ↓
所有 Text 重新排版 → 重新生成 Mesh → Canvas 重新 Batch
```

**关键**：不是"变了一个 Text"。界面有 50 个 Text 用同一动态字体 → 图集一次更新 → 50 个 Text 全部重建 → Canvas 整批刷新。

这就是聊天界面首次打开卡顿的原因：动态图集首次遇到几十上百个新汉字 → 密集栅格化 + 图集写入 → 可能触发多次 textureRebuilt → 所有 Text 反复重建。

**Font.RequestCharactersInTexture()** 是 Unity C# 层主动请求字符生成的 API：

```csharp
font.RequestCharactersInTexture("你好世界");
```

调用后 native 层（FreeType）执行栅格化，结果写入字体图集。`Font.GetCharacterInfo()` 获取单个字符的 UV、Advance 等信息——Text 组件最终基于这些数据生成 Mesh。

## 15.3 UGUI Text 的渲染流程

### 15.3.1 Text 在 Graphic 体系中的位置

```
Graphic
  └── MaskableGraphic
       └── Text
```

Text 和 Image 是兄弟——都继承 MaskableGraphic，都走 `Graphic.OnPopulateMesh() → CanvasRenderer.SetMesh() → Canvas.BuildBatch() → GPU`。区别在 **OnPopulateMesh 里做了什么**：Image 拿 Sprite 的 UV 拼一个 Quad，Text 走一整条"字符串→排版→顶点"链路。

### 15.3.2 修改 text 后：延迟刷新

```csharp
text.text = "Hello";
```

不是立刻渲染——UGUI 使用延迟刷新：

```
text.text = "Hello"
  → SetVerticesDirty()    // 标记顶点需要重新生成
  → SetLayoutDirty()      // 标记布局尺寸可能变化
    → CanvasUpdateRegistry.RegisterCanvasElementForGraphicRebuild(this)
      → 等下一帧 Canvas 统一调用 Rebuild()
```

一帧内改三次 text，只执行一次重建。

### 15.3.3 TextGenerator.Populate()：整个链路的核心

Canvas 下一帧 Rebuild 时，`Graphic.Rebuild() → UpdateGeometry() → TextGenerator.Populate()`：

```
TextGenerator.Populate(字符串)
  ├── ① 逐字符遍历
  │     检查每个字符是否在字体图集中
  │     不在 → 请求动态字体生成（栅格化 + 写入图集）
  │
  ├── ② 排版计算
  │     累加 Advance 确定每个字符的 X 位置
  │     判断是否超出宽度 → 决定换行
  │     根据对齐方式调整整行偏移
  │
  └── ③ 生成顶点（每个字符一个 Quad）
        4 顶点 + 2 三角形 + UV（从图集取字符区域）
        + Vertex Color（text.color）
          → 写入 VertexHelper → 生成 Mesh → CanvasRenderer.SetMesh()
```

每个字符 = 1 个 Quad = 4 顶点 + 2 三角形。100 字 = 400 顶点——比 Image 的 4 顶点大了 100 倍。**这是文本 CPU 开销的根源。**

### 15.3.4 文本合批

Text 合批和 Image 规则相同——同一个 Material 实例 + 同一张 Texture：

```
Text A → Arial 图集 → 材质 M
Text B → Arial 图集 → 材质 M
→ 同一材质 + 同一纹理 → 可以合批 ✅

Text A → Arial 图集
Text B → 宋体图集
→ 不同纹理 → 无法合批 ❌
```

动态图集更新后 → 图集纹理变了 → 材质引用可能变 → Batch 被打断。这就是文本系统是 DrawCall 高频来源的原因。

### 15.3.5 Best Fit 和 Rich Text 的代价

- **Best Fit**（自动缩放）：暴力循环——从大字号往小试，每次完整执行一次 Populate()。一个字号的尝试已经贵了，循环 N 次是 N 倍。大型项目几乎都禁用。
- **Rich Text**（`<b>`, `<color>`, `<size>`）：Populate 阶段需额外解析标签、维护样式栈（当前字号/颜色/粗细）。嵌套越多越贵。

## 15.4 TextMeshPro：SDF 字体 + 独立文本引擎

### 15.4.1 根本区别：SDF 取代位图

传统 UGUI Text 字体图集存**像素颜色**——字符被栅格化成固定分辨率的位图。放大 3 倍 → 像素拉伸 → 模糊。

TMP 字体图集存 **Signed Distance Field（SDF，有符号距离场）**——每个像素存的是**到字符边缘的距离**，不是颜色：

```
传统位图纹理：
  [黑][黑][黑][灰][白][白]  ← 存颜色
  放大后：块状锯齿

SDF 纹理：
  [-3][-2][-1][0][+1][+2][+3]  ← 存距离
  Shader 实时算："距离 > 0.5 → 在字符外 → 透明"
  放大后：边缘平滑（距离场可平滑插值）
```

边缘信息是 Shader 从距离场实时计算的，不固化在纹理里，放多大都不锯齿。这就是 TMP 文本质量远超传统 Text 的根本原因。

**SDF 的附加福利**：描边/阴影/发光/柔化全部在同一个 Shader 完成，不需要额外 DrawCall。传统 Text 做描边要渲染多层文本叠加。

### 15.4.2 TMP 的核心架构

```
TMP_Text（核心基类）
  └── TextMeshProUGUI（UI 组件）
        ├── TMP_FontAsset（字体资产，预构建 SDF 图集）
        │     ├── Character Table（字符→字形索引映射）
        │     ├── Glyph Table（字形→图集 UV 映射）
        │     └── Atlas Textures（Multi Atlas，支持多张图集）
        ├── TMP_SubMeshUI（不同材质时自动生成子 Mesh）
        └── Fallback Font Assets（多级回退字体）
```

TMP 不用 Unity 的 Font 对象，用自己的 `TMP_FontAsset`——编辑阶段完成 SDF 栅格化和图集打包，运行时不需要逐字符动态生成（除非遇到不在图集中的字符）。

### 15.4.3 TMP 的动态图集为什么更稳定

- **传统 Text**：单张动态图集 → 满了就扩容/重建 → 所有 Text 全部级联刷新
- **TMP**：Multi Atlas——第一张满了自动创建第二张，旧图集不受影响。不会因为加一个字符把整张图集打乱重排

### 15.4.4 TMP 的代价

- **顶点数据更多**：每个字符需额外存 Normal、Tangent 等信息（供 SDF Shader 用）
- **SDF Shader 更重**：传统 UI Shader 几乎不算，SDF Shader 要读距离场、算边缘、做描边/发光
- **SubMesh 增加 DrawCall**：文本含不同材质（Sprite、Fallback Font）时自动生成 TMP_SubMeshUI → 每个 SubMesh 一个 DrawCall
- **RichText 解析开销**：TMP 支持更丰富的标签（`<link>`, `<sprite>`, `<gradient>`），解析成本更高

## 15.5 TMP vs 传统 Text 对比

| 维度 | 传统 UGUI Text | TextMeshPro |
|------|---------------|-------------|
| 字体纹理 | 位图（像素颜色） | SDF（距离场） |
| 放大效果 | 模糊、锯齿 | 锐利（边缘实时计算） |
| 动态图集 | 单张，扩容/重建影响所有 Text | Multi Atlas，新建不影响旧图集 |
| 文本特效 | 需额外 DrawCall 或组件 | Shader 内完成（Outline/Glow/Shadow） |
| 排版能力 | 基础换行 + 对齐 | Unicode Line Breaking、Kerning、多语言 |
| CPU（单次更新） | 较低 | 较高（顶点更多、RichText 解析更重） |
| GPU | 较低（简单 Shader） | 较高（SDF Shader） |
| 动态图集稳定性 | 差（textureRebuilt 级联刷新） | 好（Multi Atlas 隔离） |
| 适合场景 | 旧项目兼容、极轻量 UI | 现代项目、高清 UI、国际化、富文本 |

**核心结论**：TMP 的单次文本更新成本比传统 Text 高，但动态图集的全局稳定性好得多。大多数项目中，级联刷新带来的帧率抖动比单次成本差异更致命——所以 TMP 更受青睐。

## 15.6 文本性能优化实战

1. **字体预热**：启动时预先请求常用字符，避免运行时首次遇到触发动态生成
   ```csharp
   font.RequestCharactersInTexture("0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ你好世界开始退出设置");
   ```

2. **避免高频 text 赋值**：不要每帧改 text。需要实时刷新数值时，先判断内容是否真的变了再赋值

3. **禁用 Best Fit**

4. **减少 Rich Text 嵌套深度**

5. **动态文本区域单独 Canvas**：防止一次 text 更新触发整个界面 Canvas 重建

6. **TMP 的 Sprite/Emoji**：注意 SubMesh 带来的额外 DrawCall，大量使用需评估

## 本章总结

```
文本渲染的核心链路：

字符串
  → 逐字符解析 Unicode
    → 查字体文件获取 Glyph 索引
      → 查字形图集获取 UV（不在图集 → 动态栅格化 → 写入图集 → 可能触发 textureRebuilt 级联）
        → 排版计算（Advance 累加 + 换行 + 对齐）
          → 生成每字符 Quad（4 顶点 × N 字符）
            → VertexHelper → Mesh → CanvasRenderer.SetMesh()
              → Canvas.BuildBatch() → GPU
```

文本系统不是"另一种 Image"——它是 UGUI 中 CPU 开销最大的子系统。原因就是**顶点数**：Image 固定 4 个顶点，Text 的顶点数 = 字符数 × 4，还要加上字符解析和排版计算的 CPU 时间。

**传统 Text → TMP 的演进本质**：把"像素颜色"换成"距离场"，把"单张动态图集"换成"Multi Atlas"，把"位图拉伸模糊"换成"Shader 实时算边缘"。代价是单次更新更贵、Shader 更重，但换来了全局稳定性——不会因为一个新字符把整个界面拖垮。
