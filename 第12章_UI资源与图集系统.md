# 第12章 UI 资源与图集系统

> 本文是对原文的结构化重写，按自底向上的顺序重新组织（Texture → Sprite → Atlas → Dynamic Atlas → 生命周期），合并原文大量重复内容。对照 UGUI 源码修正了 4 处错误。

---

## 概述

UGUI 的渲染效率不单纯取决于组件数量或层级复杂度，底层资源组织方式才是性能瓶颈的关键。本章核心论点：

```
Sprite（逻辑层）→ 描述"用纹理的哪一块"
Texture2D（数据层）→ 真正的 GPU 像素载体
SpriteAtlas（静态图集）→ 构建时合并，运行时替换，主力合批方案
动态图集 → 运行时临时合并无法提前进图集的纹理，补充方案
UI 生命周期 → 决定资源何时加载/驻留/释放
```

---

## 12.1 Sprite 与 Texture2D 的关系

### 12.1.1 核心区分

| | Texture2D | Sprite |
|------|------|------|
| 本质 | GPU 可采样的像素数组 | 对 Texture2D 某区域的描述 |
| 存储 | RGBA 像素、Mipmap、压缩数据 | Rect、UV、Pivot、Border、Texture 引用 |
| 内存 | 大（2048² RGBA32 ≈ 16MB） | 极小（几十字节） |
| GPU 可见 | ✅ | ❌（GPU 只认识 Texture） |
| 等价说法 | "数据层" | "规则层" |

这个区分是整个图集系统成立的前提——正因为多个 Sprite 可以指向同一张 Texture2D 的不同区域，图集才有合并的可能。

### 12.1.2 Sprite 的关键属性

```csharp
// Sprite 核心字段
sprite.texture    // 引用哪张 Texture2D（运行时可能被 SpriteAtlas 替换）
sprite.rect       // 在 Texture 中的像素区域 → 用于生成 UV
sprite.pivot      // 中心点 → 影响对齐、旋转、Sliced 拉伸行为
sprite.border     // 九宫格边界 → Image.Type = Sliced 时生效
```

**运行时注意**：当 Sprite 被打入 SpriteAtlas 后，`sprite.texture` 返回的是 Atlas Texture 而非原始纹理——这个替换是图集系统在运行时自动完成的，调用方无感知。

### 12.1.3 Image 为什么用 Sprite 而非 Texture

Image 需要的不只是"显示一张图"，还需要：

- 自动生成 UV（依赖 `sprite.rect`）
- 九宫格切片（依赖 `sprite.border`）
- 图集自动打包（依赖 Sprite 的纹理引用替换机制）
- Pivot 对齐和 Mesh 生成

Texture2D 只提供像素，不具备这些规则信息。所以 Image 用 Sprite，RawImage 用 Texture——前者的能力是后者的超集。

### 12.1.4 RawImage 与 SpriteAtlas 的关系

RawImage 直接引用 `Texture`，绕过了 Sprite 体系，因此**不享受 SpriteAtlas 的自动纹理替换**。这意味着即使它和其他 Image 用了"同一张图"（比如它们恰好来自同一 Atlas），SpriteAtlas 也不会自动帮 RawImage 做绑定——RawImage 合批只能靠手动确保使用相同 Texture。

**使用场景**：视频输出、摄像机 RenderTexture、动态下载的图片。普通 UI 应优先使用 Image + Sprite。

### 12.1.5 Sprite 的 Mesh 生成：Full Rect vs Tight

Sprite 导入设置中有两种 Mesh 类型：

| | Full Rect | Tight |
|------|------|------|
| 顶点 | 4（矩形） | 不定，贴合 Alpha 形状 |
| Overdraw | 透明区域也参与绘制 | 大幅减少 |
| CPU | 低 | 较高 |
| 适用 | 矩形 UI、高频更新 | 不规则形状、静态 UI |

**一个重要澄清**：Tight Mesh 增加的是顶点数量，但顶点数不影响 UGUI 合批——`Canvas.BuildBatch()` 按渲染状态（Material/Texture/Shader）分组，不按顶点数分组。Tight Mesh 的真正代价在 CPU 顶点生成和潜在的重建成本。

### 12.1.6 Sprite.Create 运行时创建

```csharp
Sprite.Create(tex, new Rect(0, 0, 128, 128), new Vector2(0.5f, 0.5f));
```

这只创建一个区域描述对象，**不复制纹理数据**，开销很小。真正昂贵的是 `Texture2D` 上传 GPU（`Apply()`），不是 Sprite 创建本身。

### 12.1.7 内存误区

**一张 2048×2048 RGBA32 纹理 = ~16MB，无论切出 10 个还是 1000 个 Sprite，纹理内存始终是 16MB。** Sprite 只占几十字节的元数据。项目中的内存问题是 Texture 过多/过大，不是 Sprite 太多。

## 12.2 SpriteAtlas 静态图集系统

### 12.2.1 本质：构建时合并 + 运行时替换

SpriteAtlas 文件本身是个**配置资产**，不是图片。它记录：哪些 Sprite 属于这个图集、用什么压缩格式、Padding 多少。真正的图集纹理在构建时生成。

运行时的核心机制：

```
Sprite（原始引用独立纹理）
  → 运行时检测到属于 SpriteAtlas
    → sprite.texture 被替换为 Atlas Texture
      → sprite.rect/UV 重新映射到 Atlas 中的位置
        → UGUI 无感知地使用 Atlas Texture 渲染
```

**开发者引用的始终是原始 Sprite，但 GPU 采样的是 Atlas Texture 的对应区域。** 不需要改代码，不需要改 Image.sprite。

### 12.2.2 为什么图集能减少 DrawCall

GPU 在一次 DrawCall 中绑定 Texture 后，可以连续绘制多个使用同一 Texture 的图元。切换 Texture = 状态切换 = Batch 中断。

```
无图集：Image A (Tex1) → Image B (Tex2) → Image C (Tex3)
        DrawCall 1       DrawCall 2       DrawCall 3     ← 三次纹理切换

有图集：Image A (Atlas) → Image B (Atlas) → Image C (Atlas)
        └──────────── DrawCall 1 ─────────────┘           ← 一次
```

**图集减少的不是图片数量，而是 GPU 纹理绑定切换次数。**

### 12.2.3 合批的其他条件（Texture 不是唯一条件）

即使 Texture 相同，以下条件不同也会打断 Batch：

| 条件 | 说明 |
|------|------|
| **Material** | 不同 Material 实例 = 不同渲染状态 |
| **Shader** | 不同 Shader = 不同 GPU 程序 |
| **Canvas** | 不同 Canvas 之间默认不合批（Canvas 是 Batch 边界） |
| **Mask (Stencil)** | Mask 修改 Stencil 状态 → 强制断批 |
| **RectMask2D** | 通过 `CanvasRenderer.EnableRectClipping()` 修改裁剪状态 → 可能断批 |
| **渲染顺序** | 被不同纹理/材质元素分隔时，同纹理元素也可能分到不同 Batch |

所以**图集提升的是合批概率，不是合批保证。** 诊断工具是 Frame Debugger。

### 12.2.4 Padding：被低估的关键参数

图集中 Sprite 之间需要额外空白像素（Padding），原因是**GPU 纹理采样会读取相邻像素**：

- 双线性过滤（Bilinear）：采样一点会混合周围 4 个像素
- Mipmap 低级别：压缩后一个采样点可能覆盖大面积
- 旋转/缩放：UV 轻微偏移就会越界

无 Padding → 按钮 A 的边缘可能染上图标 B 的颜色 → "串色"。**Padding 不是浪费空间，是纹理采样的隔离带。**

### 12.2.5 Tight Packing vs Rectangle Packing

| | Rectangle | Tight |
|------|------|------|
| 排列方式 | 完整矩形 | 按不透明区域裁剪 |
| 图集利用率 | 较低 | 较高 |
| Sprite Mesh | 始终矩形 | 可能变复杂 |
| Shader 兼容性 | 无问题 | 部分 Shader 可能异常 |

大型 UI 项目大多用 Rectangle Packing——稳定性优先于空间利用率。

### 12.2.6 图集拆分策略（实战）

**一张超级大图集的代价**：

- 打开一个界面就要加载整张 4096² 图集（~16MB GPU 显存）
- 热更新粒度粗——改一个小图标要重下整张图集
- 浪费带宽和内存

**按功能拆分的经验法则**：

| 图集 | 内容 | 特点 |
|------|------|------|
| 通用 UI | 按钮、弹窗、通用图标 | 几乎每个界面都用到，常驻 |
| 按界面/模块 | 商城、战斗、活动 | 只在对应界面打开时加载 |
| 高清/低清 Variant | 同一套 Sprite 的不同质量版本 | 设备适配 |

核心原则：**让同一个界面中同时渲染的 UI 使用同一张图集。** 不同界面之间的纹理不需要合并——反正它们不同时渲染。

### 12.2.7 SpriteAtlas 与 AssetBundle

Sprite 资源与 Atlas Texture 是分离的——开发者引用的是 Sprite（在 Bundle A），实际渲染需要 Atlas Texture（在 Bundle B）。

**常见事故**：Bundle A 加载了但 Bundle B（Atlas）没加载 → Sprite 找不到图集 → **回退到原始纹理** → DrawCall 激增。

解决：确保 Atlas Bundle 在 Sprite Bundle 之前加载，且生命周期覆盖所有引用该图集的 UI。

### 12.2.8 Variant Atlas

同一套 Sprite 生成不同质量版本的图集：高清（RGBA32）、中清（ASTC 6×6）、低清（ASTC 8×8）。**不只是缩放，还涉及压缩格式和压缩质量的变化。** 用于设备适配和热更新分包。

### 12.2.9 Include In Build

- **勾选**：图集打进包体，无需额外加载。适用于常驻核心 UI。
- **不勾选**：图集不进包。适用于 Addressables/AssetBundle 热更新。**注意**：如果包体和热更新各有一份图集 → 纹理重复 → 内存翻倍。

## 12.3 动态图集

> ⚠ **注意**：动态图集（DynamicAtlas）是 Unity 引擎内部实现，不在开源 UGUI 仓库中。以下基于官方文档和可观察行为总结，与静态图集部分不同，无法逐行对照源码验证。

### 12.3.1 与 SpriteAtlas 的本质区别

| | SpriteAtlas | 动态图集 |
|------|------|------|
| 时机 | 构建时离线生成 | 运行时按需创建 |
| 稳定性 | 高，结果可预期 | 受运行时状态影响，行为不总是可预测 |
| CPU 成本 | 零运行时开销 | 纹理复制 + UV 重映射 = 额外 CPU |
| 适用资源 | 提前准备好的 Sprite | 动态下载的、运行时生成的纹理 |
| 角色 | 主力方案 | 补充方案 |

### 12.3.2 工作流程

```
运行时检测到一个小纹理
  → 检查是否满足条件（格式、尺寸、无 Mipmap 等）
    → 在现有动态图集中找空位
      → 有空位：复制像素到图集 → 重映射 UV → 渲染时用图集纹理
      → 无空位：创建新动态图集 → 重复上述
      → 不满足条件：跳过，用原始纹理渲染
```

**关键**：原始纹理不会被删除或修改。动态图集只是额外创建了一张合并纹理作为渲染优化缓存。

### 12.3.3 哪些纹理可以进入动态图集

必要条件（粗略，具体取决于 Unity 版本）：

- **尺寸**：通常 ≤ 256×256 或 ≤ 512×512（取决于 Unity 版本和 Canvas 设置）
- **格式**：必须是常见的未压缩格式（如 RGBA32、ARGB32），Crunch 压缩的纹理不行
- **无 Mipmap**
- **非 RenderTexture**
- **Read/Write 未开启的情况下也可能被拒绝**（Unity 需要 CPU 端读取像素来复制）
- **2D Texture**（非 Cubemap、非 3D Texture）

### 12.3.4 动态图集的代价

**CPU 开销**：检查纹理状态 + 分配空间 + 拷贝像素 + 重建 UV + 更新 Batch。**页面首次打开时的卡顿往往与动态图集创建有关**——大量小纹理同时被复制到图集。

**内存**：动态图集占用额外的 GPU 显存（一张 1024² RGBA32 = 4MB），而且可能存在多张。如果纹理已经在 CPU 内存中，再复制到动态图集相当于 GPU 端也存了一份。

**碎片**：频繁创建/扩容/销毁 → GPU 内存碎片。动态头像、聊天表情、富文本内嵌图是重灾区。

### 12.3.5 动态图集失效 ≠ 合批失败

这是一个重要的概念区分：

| 问题 | 原因 | 诊断方式 |
|------|------|---------|
| 纹理没进入动态图集 | 格式不兼容、尺寸超标、有 Mipmap | Frame Debugger 中看不到 Dynamic Atlas |
| 纹理进了动态图集但没合批 | Material 不同、Mask 打断、Canvas 不同 | Frame Debugger 看到 Dynamic Atlas 但仍有多个 DrawCall |

**原文将两者混在一起列——这是不对的。** 先判断纹理是否进入动态图集，再判断是否成功合批，是两步独立排查。

### 12.3.6 实战建议

- **小型项目/独立开发者**：依赖动态图集可以省去手动管理图集的成本
- **大型商业项目**：以 SpriteAtlas 为主，动态图集只作为运行时补充。通过预热（提前打开一次界面触发动态图集生成）减少首次卡顿，同时严格控制动态纹理的数量和尺寸

## 12.4 资源加载与 UI 生命周期

### 12.4.1 UI 打开时实际发生了什么

```
① 资源加载
  加载 Prefab → 递归加载依赖（SpriteAtlas、Texture、Material、Font、Shader）

② GameObject 实例化
  Instantiate → Awake → OnEnable → Graphic 注册到 GraphicRegistry
  → CanvasUpdateRegistry 注册 Layout/Graphic 重建

③ Layout Rebuild
  ILayoutElement → ILayoutController → 递归计算 → SetDirty

④ Graphic Rebuild
  OnPopulateMesh() → CanvasRenderer.SetMesh()

⑤ 首次渲染
  Canvas.BuildBatch() → 上传 GPU → 动态图集创建（如有）→ DrawCall

⑥ 后续帧
  仅脏元素参与重建
```

**首次打开卡顿的根源**：步骤①（Shader 编译 + Texture 上传 GPU）和步骤⑤（动态图集拷贝像素）是 CPU 最密集的两段。

### 12.4.2 SetActive(false) ≠ 释放资源

```
gameObject.SetActive(false)
  → 停止 Update / 不参与渲染 / 不参与 Raycast
  → 但：Texture 仍在内存、Atlas 仍在内存、Material 未销毁
  → Prefab 实例仍持有所有引用
  → 只要引用链不断，资源就不会释放
```

**隐藏 UI = 隐藏渲染，不是释放内存。** 这是项目内存持续膨胀的最常见原因。

### 12.4.3 Destroy 与资源释放

```
Destroy(gameObject)
  → GameObject 销毁、Component 移除
  → 引用链断开（仅当没有其他引用时）
  → 资源进入"无引用"状态
  → 下次 Resources.UnloadUnusedAssets() 时真正释放
```

**Destroy 是立即执行的，但资源释放是延迟的。** 而且 Destroy 本身有 GC 开销（大量 GameObject 销毁 = 大量 GC Alloc）。

### 12.4.4 Resources.UnloadUnusedAssets

遍历所有托管对象 → 标记无引用的资源 → 释放 Texture、Mesh、Material、AudioClip 等。

**代价**：遍历整个资源引用图，是重量级操作，通常造成明显卡顿。大型项目一般只在加载界面调用，不会频繁使用。

### 12.4.5 SpriteAtlas 为什么不会随 UI 一起释放

Atlas 是**共享资源**：主界面和商城界面可能引用同一张图集。关闭主界面 ≠ 图集没人在用 ≠ 不能释放。

**Atlas 的生命周期应该独立于 UI 生命周期来管理：**

- 常驻图集 → 永不释放
- 模块图集 → 模块退出时确认无引用后释放
- 活动图集 → 活动结束后释放

### 12.4.6 UI 缓存池的取舍

```
Instantiate/Destroy 模式：
  创建开销大（GC + Rebuild）→ 但内存可控

缓存池模式：
  首次创建后常驻 → 打开/关闭几乎零开销 → 但资源长期占用内存
```

**策略**：高频 UI（主界面、背包、HUD）→ 缓存池。低频 UI（设置、公告、一次性引导）→ 正常 Destroy。

### 12.4.7 Addressables 与 UI

Addressables 使用引用计数管理资源：

```
加载 Prefab → Handle 引用计数 +1
  → 使用完毕 → Addressables.Release(handle) → 引用计数 -1
    → 计数归零 → 资源允许被释放
```

**常见错误**：只 Destroy 了 GameObject，没 Release Handle → AssetBundle 永远无法卸载 → 内存泄漏。**Addressables 的生命周期管理本质上是 Handle 的生命周期管理。**

### 12.4.8 GPU 纹理上传的成本

CPU 侧纹理内存（系统 RAM）和 GPU 侧纹理内存（显存）是两份：

- 一张 2048² RGBA32 纹理 ≈ 16MB CPU + 16MB GPU = 32MB 总计
- `Texture2D.Apply()` = CPU → GPU 上传通道，耗时操作
- 频繁加载/卸载 UI → 反复上传纹理 → GPU Stall → 掉帧

**对策**：核心图集预加载常驻，非核心图集按需加载但避免高频重复。

## 本章总结

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  逻辑层                     数据层                           │
│  Sprite（区域描述）  ──→  Texture2D（像素载体）               │
│       │                        │                            │
│       │  ┌── SpriteAtlas（静态，构建时合并）──┐              │
│       └──┤  动态图集（Runtime，临时合并）    ├──→ GPU         │
│          └──────────────────────────────────┘               │
│                                                             │
│  UI 生命周期 = 资源引用链的生命周期                           │
│  打开 ≠ 加载完毕，隐藏 ≠ 释放，Destroy ≠ 立即释放             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

三个核心结论：

1. **Sprite 是规则层，Texture2D 是数据层**——图集合并的可行性建立在这个分离之上
2. **图集减少的是 GPU 纹理切换次数**——合批概率提升，但不是合批保证（Material、Canvas、Mask 也是合批条件）
3. **UI 性能问题的根源往往不在组件代码，而在资源组织方式和生命周期管理**
