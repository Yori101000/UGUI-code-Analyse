## 概述

前三章分别讲了怎么构建顶点（第 16 章）、怎么通过 Shader 渲染（第 17 章）、怎么实现特效（第 18 章）。本章反过来——从性能角度审视这些操作到底有什么代价。

```
UGUI 性能问题不是单一模块导致的，而是整个渲染体系共同叠加的结果。
一个 Outline 组件可能让 1 个 DrawCall 变成 3 个，
一个 Mask 嵌套可能让 GPU FillRate 翻倍，
一个 graphic.material 赋值可能让共享材质全部实例化。
```

UI 性能问题可以归纳为五个维度：

| # | 问题 | 影响谁 | 诊断工具 |
|---|------|--------|---------|
| 1 | **Batch 中断** | CPU（DrawCall） | Frame Debugger → Batch Breaking Reason |
| 2 | **材质实例化** | CPU（DrawCall）+ 内存 | Frame Debugger → Different Material |
| 3 | **Mask/Stencil 状态** | CPU（材质实例）+ GPU（Fragment） | Frame Debugger → Different Stencil State |
| 4 | **顶点膨胀** | CPU（Canvas Rebuild）+ GPU（Vertex） | Profiler → Canvas.BuildBatch, Stats → Verts |
| 5 | **Overdraw** | GPU（FillRate） | Scene View → Overdraw Draw Mode |

## 19.1 Batch 中断

### 合批条件：精确匹配

UGUI 的 Batch 合并条件是**精确匹配**，不是"相似即可"：

```
两个 UI 元素必须拥有完全一致的：
  ├── Material（同一个 C# 对象实例）
  ├── Texture（同一张纹理）
  ├── Shader（同个 Shader + 同组 Keyword）
  ├── Stencil 状态（Ref / Comp / Pass 相同）
  ├── RectMask2D 裁剪矩形（相同或都未设置）
  ├── Canvas（属于同一个 Canvas）
  └── Additional Shader Channels（一致）
```

只要其中任意一项不同 → Batch 中断 → 新 DrawCall。

### 渲染顺序的影响

UGUI 是**严格层级渲染系统**，渲染顺序由 Hierarchy 中的节点顺序决定，不会为了合批打乱顺序：

```
Hierarchy：A（材质 M1）→ B（材质 M2）→ C（材质 M1）
结果：Batch 0 → A | Batch 1 → B | Batch 2 → C
```

即使 A 和 C 材质相同，它们也无法合并。因为 Unity 为了保证 UI 层级正确，必须先画 A、再画 B、最后画 C。

**实操建议**：把相同材质的 UI 元素在 Hierarchy 中排在一起——可以减少中间被不同材质打断的次数。

### Canvas 是 Batch 的天然边界

不同 Canvas 之间的 UI 即使材质、纹理完全相同，也分别提交独立 DrawCall。

但同一个 Canvas 太大会导致另一个问题：

```
超大 Canvas：任意子节点变化 → 整个 Canvas 重建 → 所有 Mesh 重新生成
多个小 Canvas：DrawCall 增加（每个 Canvas 独立提交）
```

**解决方案**：按更新频率拆分 Canvas：

| Canvas | 内容 | 更新频率 |
|--------|------|---------|
| 静态 Canvas | 背景、装饰、Logo | 几乎不变 |
| 动态 Canvas | 血条、数字、倒计时 | 中等频率 |
| 高频 Canvas | 聊天、实时数值 | 频繁更新 |

这样即使高频 Canvas 不断重建，也不会影响静态 Canvas 的 Mesh。

## 19.2 材质实例化

### 两个属性的真正区别

```csharp
// material（getter）：如果当前材质是共享材质，自动 Instantiate() 生成新实例再返回
// 适合：需要单独修改这个 Image 的颜色，又不影响其他使用同材质的 Image
var mat = image.material;        // ← 内部触发 Instantiate（如果之前是共享材质）
mat.color = Color.red;           // 只改这一个 Image 的材质

// sharedMaterial（getter）：直接返回引用，不会实例化
// 适合：只读材质参数，或确认你想全局修改
var mat = image.sharedMaterial;  // ← 不触发 Instantiate
mat.color = Color.red;           // ← 所有使用这个材质的 UI 全部变色
```

**关键对比**：

| | `material` | `sharedMaterial` |
|------|-----------|-----------------|
| getter 返回 | 实例化后的独享材质 | 共享材质原引用 |
| 修改属性 | 只影响这一个 Graphic | 影响所有引用者 |
| 触发实例化 | ✅ 自动（如果当前是共享的） | ❌ 不会 |
| 适合场景 | 需要单独修改颜色/参数 | 只读读取、或全局修改 |

```csharp
// 正确用法示例

// ① 只读材质参数 → sharedMaterial（不触发实例化）
Color c = image.sharedMaterial.color;

// ② 单独修改一个 Image 的颜色 → material（触发一次实例化，但之后安全）
image.material.color = Color.red;

// ③ 批量修改多个 Image → sharedMaterial + 一次赋值（所有 UI 同时变）
image.sharedMaterial.color = Color.red;  // 所有使用该材质的 UI 一起变

// ④ 危险：每帧触发实例化
void Update()
{
    image.material.color = Color.Lerp(a, b, t);  // ❌ 每帧 new Material()
}
```

`graphic.material` 的 getter 在检测到材质可能被修改时，会调用 `Graphic.SetMaterialDirty()` 并创建材质的新实例。**在 Update 中频繁访问 `graphic.material` 是 UGUI 最常见的性能陷阱之一。**

### 哪些 UI 特效会隐式创建材质实例

| 场景 | 原因 | 诊断标志 |
|------|------|---------|
| Mask 组件 | `StencilMaterial.Add()` 生成带 Stencil 参数的新材质 | Frame Debugger → Different Stencil |
| 第三方特效插件 | 内部 `new Material(baseMaterial)` | 不同 Material 实例 |
| Outline / Shadow | 内置组件不创建新材质（通过 ModifyMesh 实现） | 不涉及材质变化 |
| 运行时改 Shader 参数 | `material.SetFloat()` 触发材质实例化 | Material 数量增加 |

### Shader Keyword 导致的隐式断批

同一个 Shader 文件，但不同的 Keyword 组合，在 GPU 视角中是不同的 Shader Variant：

```
Image A → UNITY_UI_CLIP_RECT ON
Image B → UNITY_UI_CLIP_RECT OFF
```

即使它们引用同一个 `ui/default` Shader 文件，Batch 也会被打断——GPU 加载的已经是两个不同的着色器程序。

## 19.3 Mask/Stencil 的代价

> 这一节和第 13 章（Mask 裁剪机制）内容重叠。本章只从性能维度补充关键数据。

### Mask 增加 DrawCall 的原因链

```
Mask 组件挂上
  → StencilMaterial.Add() 创建新材质实例（带 Stencil 参数）
    → Mask 自身 Image 使用这个材质
      → 子节点使用另一个 Stencil 材质（不同 Stencil 参数）
        → 外部的 UI 使用普通材质（没有 Stencil）
          → 三个不同材质实例 → 至少 3 个 Batch
```

每层嵌套 Mask 增加一组新的 Stencil 参数 → 每个参数组合对应一个新的材质实例 → Batch 按 Stencil 深度分裂。

### Mask 不减少 Overdraw

被 Mask 裁剪掉的像素**仍然执行了完整的 Fragment Shader**。Stencil 测试发生在 Fragment Shader 之后，丢弃决定不影响已经完成的 Fragment 运算。

| 阶段 | 是否执行 |
|------|---------|
| Vertex Shader | ✅ 执行 |
| Fragment Shader | ✅ 执行（结果被丢弃） |
| Stencil Test | 之后 → 丢弃 |
| Blend | 不执行 |

所以 Mask 不会降低 GPU FillRate 压力——该处理的 Fragment 一个没少。

### RectMask2D vs Mask

RectMask2D 不依赖 Stencil，通过 `CanvasRenderer.EnableRectClipping()` 实现裁剪，不产生材质实例化，不影响 Batch。凡是矩形裁剪场景，优先用 RectMask2D。

## 19.4 顶点膨胀

UGUI 是 Mesh 渲染系统。顶点数量直接影响：

| 阶段 | 影响 |
|------|------|
| CPU | `OnPopulateMesh`、`ModifyMesh`、`VertexHelper` 操作 |
| CPU | `Canvas.BuildBatch()` 处理大量顶点 |
| CPU→GPU | 大量顶点数据上传 |
| GPU | Vertex Shader 处理 |

### 顶点膨胀的乘法效应

```
Text（1000 字符 = 4000 顶点）
  + Outline（×5）= 20000 顶点
    + Shadow（×2）= 40000 顶点
```

Outline 的 ×5 是对**已有全部顶点**做 4 方向复制（左右上下），不是对原始顶点。

### 过量顶点的诊断标准

| 顶点数 | 风险 | 典型场景 |
|--------|------|---------|
| < 2K | ✅ 安全 | 简单 UI |
| 2K ~ 10K | ⚠️ 注意 | 少量 Text |
| 10K ~ 50K | ❌ 严重 | 大量 Text + Outline |
| 50K+ | 🔴 灾难 | 长列表 + 多层特效 |

在 Stats 面板的 `Verts` 窗口看到超过 10K 就要排查了。

## 19.5 Overdraw

### 为什么 UI 天然容易 Overdraw

3D 不透明物体可以利用深度测试提前剔除被遮挡像素（Early-Z），被遮挡的 Fragment Shader 根本不会执行。但 UI 使用透明混合（`Blend SrcAlpha OneMinusSrcAlpha`），无法使用 Early-Z——GPU 必须执行 Fragment Shader，然后才能做 Blend 决定最终颜色。

```
全屏半透明遮罩 → 每个像素一次 Fragment
  弹窗背景 → 第二次（弹窗区域）
    按钮  → 第三次
      文字 → 第四次
```

### 最危险的 Overdraw 来源

| 来源 | 危害程度 | 原因 |
|------|---------|------|
| 全屏半透明遮罩 | 🔴 极高 | 覆盖所有像素，每次整屏 Fragment |
| UI Blur | 🔴 极高 | GrabPass + 双 Pass 模糊，全屏 Fragment |
| 大面积发光/Glow | 🟡 高 | 81 次采样的代价 × Overdraw |
| 多层半透明叠加 | 🟡 高 | 弹窗套弹窗，每层都混合 |
| Text 叠 Outline | 🟡 中 | 透明区域大，字符边缘重复采样 |

### 诊断 Overdraw

Scene View → Draw Mode → Overdraw：
- 黑色 = 无 Overdraw
- 绿色 = 中度
- **红色 = 严重**
- **白色 = 爆炸（移动端必卡）**

## 优化策略优先级

```
第一优先级：减少 Canvas 重建
  按更新频率拆分 Canvas
  静态 UI 和动态 UI 分离
  避免每帧 SetVerticesDirty()

⚠️ 常见误区：只盯着 DrawCall，忽略 Rebuild 成本
  把所有 UI 合并到一个 Canvas 可以减少 DrawCall（不同 Canvas 无法合批），
  但代价是任意一个小变化都触发整个 Canvas 重建。
  DrawCall 降了，CPU 开销反而涨了。
  正确做法：在 DrawCall 和 Rebuild 之间做取舍——
  静态内容合并（省 DrawCall），动态内容拆分（省 Rebuild）。

第二优先级：减少 Batch 中断
  相同材质的 UI 在 Hierarchy 中排在一起
  统一图集、Shader、材质
  不用 graphic.material 写操作

第三优先级：控制顶点
  避免 Text + Outline + Shadow 组合
  ScrollView 中的节点复用
  少用富文本

第四优先级：降低 Overdraw
  减少全屏半透明遮罩
  Blur / Glow 限制在小范围
  移动端特别注意 Fragment 压力

第五优先级：Layout 系统
  LayoutGroup 适合编辑器排版，不适合高频动态更新
  长列表手动控制坐标
```

## 19.6 大规模项目工程优化

前 5 节讲的都是"单界面的渲染性能"。当项目有几十上百个界面时，还有一套工程层面的问题——界面切换卡顿、GC 频繁、列表滚动掉帧。这不在渲染管线的范畴，但影响更大。

### 19.6.1 UI 分层架构

大型项目通常按**界面层级类型**拆分 Canvas，而不是按"功能模块"拆：

```
UIRoot
├── BottomLayer Canvas  ← 底层主界面（常驻、很少刷新）
├── WindowLayer Canvas  ← 功能窗口（背包、商城）
├── PopupLayer Canvas   ← 弹窗层（确认框、奖励弹窗）
├── TopLayer Canvas     ← 顶层提示（Toast、飘字）
├── GuideLayer Canvas   ← 新手引导
└── LoadingLayer Canvas ← Loading 界面
```

每一层独立 Canvas，互不影响。这样底层主界面的静态 UI 不会因为弹窗打开而重建，弹窗也不会因为底层刷新而重建。**分层 = 隔离重建范围。**

拆分原则按刷新频率：

| Canvas 类型 | 内容 | 更新频率 | 典型位置 |
|-----------|------|---------|---------|
| 静态 Canvas | 背景、装饰、固定按钮 | 几乎不变 | BottomLayer |
| 动态 Canvas | 血量、金币、倒计时 | 中等频率 | 窗口层内独立 |
| 高频 Canvas | 聊天、实时排行榜 | 频繁 | 单独子 Canvas |
| ScrollView Canvas | 列表内容 | 持续变化 | 模块内独立 |
| 动画 Canvas | Tween 动画、渐变 | 每帧变化 | 独立 Canvas |

### 19.6.2 UI 生命周期管理

很多项目卡顿不是因为渲染，而是因为界面管理混乱——同一个界面重复打开、事件重复注册、关闭后资源不释放。

标准化的 UI 生命周期：

```
Create     → 实例化 Prefab，创建节点
Initialize → 绑定事件、注册监听器
Show       → 播放入场动画、刷新数据
Refresh    → 更新显示内容（数据变化时）
Hide       → 播放退场动画
Destroy    → 解绑事件、释放资源、回收入池
```

关键原则：
- **Create/Initialize 只执行一次**，Show/Hide 可以反复调用
- **Destroy 不一定是真销毁**——可以用缓存池替代
- **退出时一定要解绑事件**，否则界面对象永远无法 GC

### 19.6.3 对象池与 UI 缓存池

**UI 缓存池**（界面级别）：关闭的界面不 Destroy，放入缓存池，下次直接 Show 复用：

```
首次打开背包 → Instantiate
  关闭背包 → SetActive(false)，入池
    再次打开背包 → SetActive(true)，刷新数据，直接显示
```

避免了每次打开界面都走 Instantiate、加载资源、重建 Canvas 的完整链路。

**对象池**（列表节点级别）：适用于 ScrollView 中大量重复的 Item。

```csharp
public class UIObjectPool<T> where T : Component
{
    private Queue<T> pool = new Queue<T>();
    private T prefab;
    private Transform parent;

    public UIObjectPool(T prefab, int preloadCount, Transform parent)
    {
        this.prefab = prefab;
        this.parent = parent;
        for (int i = 0; i < preloadCount; i++)
        {
            T obj = GameObject.Instantiate(prefab, parent);
            obj.gameObject.SetActive(false);
            pool.Enqueue(obj);
        }
    }

    public T Get()
    {
        if (pool.Count > 0)
        {
            T obj = pool.Dequeue();
            obj.gameObject.SetActive(true);
            return obj;
        }
        return GameObject.Instantiate(prefab, parent);
    }

    public void Release(T obj)
    {
        obj.gameObject.SetActive(false);
        pool.Enqueue(obj);
    }
}
```

### 19.6.4 虚拟列表（Virtual Scroll）

传统 ScrollView 的问题：数据 1000 条 → 创建 1000 个 Item → 节点暴涨 → Canvas 重建爆炸 → 卡顿。

但屏幕一次只能看到 10~15 个 Item。虚拟列表的核心：**只创建可见区域的节点，滚动时循环复用。**

```
屏幕可见区：10 个 Item
系统创建： 12~15 个 Item
滚动时：
  Item 离开屏幕顶部 → 移动到屏幕底部 → 刷新数据 → 继续使用
  整个过程无 Instantiate、无 Destroy、无 GC
```

**完整实现（背包列表 1000 条数据流畅滚动）：**

```csharp
using System.Collections.Generic;
using UnityEngine;
using UnityEngine.UI;

public class BagVirtualList : MonoBehaviour
{
    [Header("UI")]
    public ScrollRect scrollRect;
    public RectTransform content;
    public BagItem itemPrefab;

    [Header("Item Setting")]
    public float itemHeight = 100f;

    private List<BagItemData> dataList = new List<BagItemData>();
    private List<BagItem> itemList = new List<BagItem>();
    private int visibleCount;
    private int lastStartIndex = -1;

    void Start()
    {
        scrollRect.movementType = ScrollRect.MovementType.Clamped;
        CreateData();
        InitList();
        scrollRect.onValueChanged.AddListener(OnScroll);
    }

    void OnDestroy()
    {
        scrollRect.onValueChanged.RemoveListener(OnScroll);
    }

    void CreateData()
    {
        for (int i = 0; i < 1000; i++)
            dataList.Add(new BagItemData
            {
                Id = i,
                Name = "物品_" + i,
                Count = Random.Range(1, 999)
            });
    }

    void InitList()
    {
        // 计算可见数量（+2 缓冲）
        visibleCount = Mathf.CeilToInt(
            scrollRect.viewport.rect.height / itemHeight) + 2;

        // 只创建可见数量的节点
        for (int i = 0; i < visibleCount; i++)
        {
            BagItem item = Instantiate(itemPrefab, content);
            item.GetComponent<RectTransform>().sizeDelta =
                new Vector2(item.GetComponent<RectTransform>().sizeDelta.x, itemHeight);
            itemList.Add(item);
        }

        // Content 高度 = 总数据量 × 每项高度（让滚动范围正确）
        content.sizeDelta = new Vector2(content.sizeDelta.x, dataList.Count * itemHeight);

        RefreshItems(true);
    }

    void OnScroll(Vector2 pos) => RefreshItems();

    void RefreshItems(bool force = false)
    {
        float scrollY = Mathf.Max(0, content.anchoredPosition.y);
        int startIndex = Mathf.FloorToInt(scrollY / itemHeight);

        if (!force && startIndex == lastStartIndex) return;
        lastStartIndex = startIndex;

        for (int i = 0; i < itemList.Count; i++)
        {
            int dataIndex = startIndex + i;

            if (dataIndex < 0 || dataIndex >= dataList.Count)
            {
                itemList[i].gameObject.SetActive(false);
                continue;
            }

            BagItem item = itemList[i];
            item.gameObject.SetActive(true);

            // 移动到正确位置
            item.GetComponent<RectTransform>().anchoredPosition =
                new Vector2(0, -dataIndex * itemHeight);

            // 刷新数据
            item.Refresh(dataList[dataIndex]);
        }
    }
}
```

**性能对比**：

| 方案 | Hierarchy 节点数 | 内存 | GC | 滚动流畅度 |
|------|:--------------:|:---:|:--:|:---------:|
| 全量创建 1000 Item | 1000+ | 高 | 频繁 | ❌ 卡顿 |
| 虚拟列表 12 节点 | ~15 | 极低 | 几乎无 | ✅ 流畅 |

**动态高度列表**（聊天消息、评论等不等高 Item）：需要额外维护高度缓存数组 + 累积高度表，滚动定位用二分查找取代除法。

## 本章总结

```
UI 性能问题的五个维度：

① Batch（DrawCall）
   材质/纹理/Shader/Stencil 任一变化 → 断批
   诊断：Frame Debugger → Batch Breaking Reason

② 材质实例化
   graphic.material / StencilMaterial.Add / new Material
   诊断：Frame Debugger → Different Material

③ Mask/Stencil
   每层 Mask = 新材质 + 断批
   Mask 不减少 Overdraw
   诊断：Frame Debugger → Different Stencil State

④ 顶点膨胀
   Outline = ×5, Shadow = ×2, Text = 4 顶点/字
   诊断：Stats → Verts > 10K → 需要关注

⑤ Overdraw
   半透明混合无法 Early-Z
   诊断：Scene View → Overdraw Draw Mode

⑥ 工程层面（大规模项目）
   UI 分层架构（按层级拆 Canvas，隔离重建范围）
   对象池 / UI 缓存池（减少 Instantiate 和 GC）
   虚拟列表（只创建可见节点，循环复用）
   生命周期管理（标准化 Create/Show/Hide/Destroy）
```
